# pi-memory — Build Spec

> Design/implementation spec. Decisions, responsibilities, interfaces, build
> order, acceptance criteria. No function bodies.

**Goal:** Persistent, write-back memory: the agent records durable learnings across sessions, and relevant memories are recalled into context — closing the gap that static rules/skills leave open.

**Architecture:** File-based store (one markdown-with-frontmatter file per memory + an index), a pure recall/format layer that produces `<pi-memory>` injections, and thin Pi glue (recall tool, write tool, session/verify hooks, config command).

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, hand-rolled minimal frontmatter parse (no dep), `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Application:
- Tools `memory_recall`, `memory_write`. Injects context wrapped in `<pi-memory>` tags.
- Emits `memory:wrote { keys }`, `memory:recalled { keys }`. Subscribes `verify:failed`.
- Enforcement dial applies as `off`/`notify` only (no blockable action — documented). Command `/pi-memory`. Log `[pi-memory]`.

## Design Decisions

1. **File-based store**, mirroring a proven memory layout: one file per fact with YAML frontmatter (`name`, `description`, `type`, plus body), a `INDEX.md` of `name — description` lines. Global at `~/.pi/agent/memory/`, project at `.pi/memory/`. Human-readable, git-friendly, no DB. (SQLite FTS5 — like `pi-hermes-memory` — is the scale alternative; flagged, not v1.)
2. **Recall = progressive disclosure, not embeddings (v1).** On `session_start` (and after compaction) inject the *index* (names + descriptions) inside a `<pi-memory>` block — always-in-context, cheap. Full bodies load on demand via `memory_recall({ name })` or `memory_recall({ query })` (keyword/substring match over name+description+body). No vector search in v1.
3. **Write-back two ways:** the agent calls `memory_write({ name, description, content, type, scope })` explicitly; and an auto-capture hook subscribes to `verify:failed` to record a gotcha (dedup by `name`). Writing an existing `name` updates in place, never duplicates.
4. **Types & scope** mirror a small fixed taxonomy: `type ∈ {user, feedback, project, reference}` (`StringEnum`), `scope ∈ {global, project}` (`StringEnum`). Scope decides which dir.
5. **Secret hygiene** — before writing, scan content for obvious secrets (API keys, tokens) and refuse/redact (pure matcher, testable). Borrowed concept from `pi-hermes-memory`.

### Open items for the human
- Whether auto-capture on `verify:failed` is on by default (recommend: on in `notify`).
- The recall relevance ceiling — how many memory bodies `memory_recall({query})` may return.

## File Structure & Responsibilities

| File | Responsibility |
|---|---|
| `src/frontmatter.ts` | pure parse/serialize of the frontmatter+body format |
| `src/store.ts` | fs: dirs (global+project), `list`, `read`, `write` (dedup), `delete`, index maintenance |
| `src/secrets.ts` | pure secret scanner → findings |
| `src/recall.ts` | pure: `formatIndexInjection`, `formatRecall`, `selectByQuery` |
| `src/tools.ts` | `memory_recall`, `memory_write` |
| `src/hooks.ts` | `session_start` index injection; `verify:failed` auto-capture |
| `src/command.ts` | `/pi-memory` (list, set mode, prune) |
| `index.ts` | factory wiring |
| `test/*.test.ts` | frontmatter, store round-trip/dedup, secrets, recall selection/format |

## Interfaces / Contracts

```typescript
type MemoryType = "user" | "feedback" | "project" | "reference";  // StringEnum
type Scope = "global" | "project";                                // StringEnum
interface Memory { name: string; description: string; type: MemoryType; scope: Scope; body: string; }

// frontmatter.ts (pure)
function parseMemory(fileText: string): Omit<Memory, "scope">;
function serializeMemory(m: Memory): string;

// store.ts
function memoryDirs(cwd: string): { global: string; project: string };
function listMemories(cwd: string): Memory[];
function readMemory(name: string, cwd: string): Memory | null;
function writeMemory(m: Memory, cwd: string): void;    // dedup by name; rewrites INDEX
function deleteMemory(name: string, cwd: string): void;

// secrets.ts (pure)
function scanSecrets(text: string): Array<{ kind: string; match: string }>;

// recall.ts (pure)
function selectByQuery(mems: Memory[], query: string, limit: number): Memory[];
function formatIndexInjection(mems: Memory[]): string;   // <pi-memory> name — description ... </pi-memory>
function formatRecall(mems: Memory[]): string;           // <pi-memory> full bodies </pi-memory>

// tools: memory_recall({ query?, name? }), memory_write({ name, description, content, type, scope })
// events: memory:wrote {keys}, memory:recalled {keys}
```

## Build Sequence

- [ ] **1 · Scaffold** — per house-style. Accept: `bun test` runs.
- [ ] **2 · Frontmatter** — Accept: `serializeMemory`→`parseMemory` round-trips name/description/type/body; tolerant of missing optional fields.
- [ ] **3 · Store** — Accept (tmp dir): `writeMemory` creates the file + INDEX entry; `writeMemory` with an existing name updates in place (no dup); `listMemories` merges global+project; `deleteMemory` removes file + index line.
- [ ] **4 · Secret scanner** — Accept: flags an OpenAI/Anthropic-style key and a `Bearer` token; ignores ordinary prose.
- [ ] **5 · Recall** — Accept: `selectByQuery` ranks/limits by keyword overlap; `formatIndexInjection`/`formatRecall` wrap in `<pi-memory>` with a `source · why` header.
- [ ] **6 · Tools** — Accept: `memory_write` refuses on secret findings, else stores + emits `memory:wrote`; `memory_recall` returns bodies + emits `memory:recalled`.
- [ ] **7 · Hooks + config** — Accept: `session_start` injects the index via the `context` channel; `verify:failed` auto-captures a gotcha in `notify` mode; `/pi-memory` lists/prunes and sets `mode`.

## Test Strategy

Everything pure (`frontmatter`, `secrets`, `recall`) fully unit-tested. `store` against a tmp dir. Tools with a fake store + captured events. Hooks verified in manual smoke (inject on start, capture on a simulated `verify:failed`).

## Risks / Notes

- Injecting the full index every session grows with memory count — cap it, and move to FTS/embeddings if it gets large (Decision 1/2).
- Auto-capture quality: a naive "record the failure" is noisy — gate it (only on repeated/again-seen failures) or keep it off by default.
- Project-scope memories in a shared repo leak into git — document that `.pi/memory/` is the user's choice to commit or ignore.
