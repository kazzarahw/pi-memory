# pi-memory тАФ Build Spec

> Design/implementation spec. Decisions, responsibilities, interfaces, build
> order, acceptance criteria. No function bodies.

**Goal:** Persistent, write-back memory: the agent records durable learnings across sessions, and relevant memories are recalled into context тАФ closing the gap that static rules/skills leave open.

**Architecture:** File-based store (one markdown-with-frontmatter file per memory + an index), a pure recall/format layer that produces `<pi-memory>` injections, and thin Pi glue (recall tool, write tool, session/verify hooks, config command).

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, hand-rolled minimal frontmatter parse (no dep), `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Application:
- Tools `memory_recall`, `memory_write`. Injects context wrapped in `<pi-memory>` tags.
- Emits `memory:wrote { keys }`, `memory:recalled { keys }`. Subscribes `verify:failed`.
- Enforcement dial applies as `off`/`notify` only (no blockable action тАФ documented). Command `/pi-memory`. Log `[pi-memory]`.

## Design Decisions

1. **File-based store**, mirroring a proven memory layout: one file per fact with YAML frontmatter (`name`, `description`, `type`, plus body), a `INDEX.md` of `name тАФ description` lines. Global at `~/.pi/agent/memory/`, project at `.pi/memory/`. Human-readable, git-friendly, no DB. (SQLite FTS5 тАФ like `pi-hermes-memory` тАФ is the scale alternative; flagged, not v1.)
2. **Recall = progressive disclosure, not embeddings (v1).** Inject the *index* (names + descriptions) inside a `<pi-memory>` block via the **`context` hook** (house-style ┬з6's channel for standing memory context) тАФ it runs on every LLM call, so the index is always present and survives compaction/forks automatically, and it's ephemeral (no queued message тЖТ no print-mode hang, unlike `session_start`+`sendMessage`). Full bodies load on demand via `memory_recall({ name })` or `memory_recall({ query })` (keyword/substring match over name+description+body). No vector search in v1.
3. **Write-back two ways:** the agent calls `memory_write({ name, description, content, type, scope })` explicitly (the primary, high-signal path); and an auto-capture hook subscribes to `verify:failed` to record a gotcha (dedup by `name`), **off by default** (naive failure capture is noisy; opt in via config). Writing an existing `name` updates in place, never duplicates (dedup across scopes too).
4. **Types & scope** mirror a small fixed taxonomy: `type тИИ {user, feedback, project, reference}` (`StringEnum`), `scope тИИ {global, project}` (`StringEnum`). Scope decides which dir.
5. **Secret hygiene** тАФ before writing, scan content for obvious secrets (API keys, tokens) and refuse/redact (pure matcher, testable). Borrowed concept from `pi-hermes-memory`.

### Resolved during implementation
- Auto-capture on `verify:failed`: **off by default** (opt in via `/pi-memory autocapture on`) тАФ naive capture is noisy.
- Recall ceiling: **3** bodies for a `query` recall (`recallLimit`, configurable).
- Index injection channel: the **`context` hook**, not `session_start` (Decision 2).
- Global dir honors `PI_CODING_AGENT_DIR` (also lets tests redirect it off the real home).

## Interfaces / Contracts

```typescript
type MemoryType = "user" | "feedback" | "project" | "reference";  // StringEnum
type Scope = "global" | "project";                                // StringEnum
interface Memory { name: string; description: string; type: MemoryType; scope: Scope; body: string; }

// frontmatter.ts (pure)
function parseMemory(fileText: string): Omit<Memory, "scope">;
function serializeMemory(m: Memory): string;

// store.ts
function memoryDirs(cwd: string): { global: string; project: string };
function listMemories(cwd: string): Memory[];
function readMemory(name: string, cwd: string): Memory | null;
function writeMemory(m: Memory, cwd: string): void;    // dedup by name; rewrites INDEX
function deleteMemory(name: string, cwd: string): void;

// secrets.ts (pure)
function scanSecrets(text: string): Array<{ kind: string; match: string }>;

// recall.ts (pure)
function selectByQuery(mems: Memory[], query: string, limit: number): Memory[];
function formatIndexInjection(mems: Memory[]): string;   // <pi-memory> name тАФ description ... </pi-memory>
function formatRecall(mems: Memory[]): string;           // <pi-memory> full bodies </pi-memory>

// tools: memory_recall({ query?, name? }), memory_write({ name, description, content, type, scope })
// events: memory:wrote {keys}, memory:recalled {keys}
```

## Build Sequence

- [ ] **1 ┬╖ Scaffold** тАФ per house-style. Accept: `bun test` runs.
- [ ] **2 ┬╖ Frontmatter** тАФ Accept: `serializeMemory`тЖТ`parseMemory` round-trips name/description/type/body; tolerant of missing optional fields.
- [ ] **3 ┬╖ Store** тАФ Accept (tmp dir): `writeMemory` creates the file + INDEX entry; `writeMemory` with an existing name updates in place (no dup); `listMemories` merges global+project; `deleteMemory` removes file + index line.
- [ ] **4 ┬╖ Secret scanner** тАФ Accept: flags an OpenAI/Anthropic-style key and a `Bearer` token; ignores ordinary prose.
- [ ] **5 ┬╖ Recall** тАФ Accept: `selectByQuery` ranks/limits by keyword overlap; `formatIndexInjection`/`formatRecall` wrap in `<pi-memory>` with a `source ┬╖ why` header.
- [ ] **6 ┬╖ Tools** тАФ Accept: `memory_write` refuses on secret findings, else stores + emits `memory:wrote`; `memory_recall` returns bodies + emits `memory:recalled`.
- [ ] **7 ┬╖ Hooks + config** тАФ Accept: `session_start` injects the index via the `context` channel; `verify:failed` auto-captures a gotcha in `notify` mode; `/pi-memory` lists/prunes and sets `mode`.

## Test Strategy

Everything pure (`frontmatter`, `secrets`, `recall`) fully unit-tested. `store` against a tmp dir. Tools with a fake store + captured events. Hooks verified in manual smoke (inject on start, capture on a simulated `verify:failed`).

## Risks / Notes

- Injecting the full index every session grows with memory count тАФ cap it, and move to FTS/embeddings if it gets large (Decision 1/2).
- Auto-capture quality: a naive "record the failure" is noisy тАФ gate it (only on repeated/again-seen failures) or keep it off by default.
- Project-scope memories in a shared repo leak into git тАФ document that `.pi/memory/` is the user's choice to commit or ignore.
